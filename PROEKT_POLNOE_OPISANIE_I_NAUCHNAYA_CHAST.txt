ПОЛНОЕ ОПИСАНИЕ ПРОЕКТА ДЛЯ ДИПЛОМНОЙ РАБОТЫ (МФТИ)
Симулятор двусторонних диалогов для голосового ассистента fast-food
================================================================================
Дата подготовки описания: 12 апреля 2026 г.
Источник фактов по коду и структуре: PROJECT_CONTEXT.md, docs/ARCHITECTURE.md,
README.md, дерево src/mcd_voice/, scripts/, tests/.

--------------------------------------------------------------------------------
1. ЧТО ЭТО ЗА ПРОЕКТ (В ДВУХ СЛОВАХ И В РАЗВЁРНУТОМ ВИДЕ)
--------------------------------------------------------------------------------

Кратко: это программный комплекс на Python, который моделирует сценарий
«drive-through» в ресторане быстрого питания (на примере меню McDonald’s):
с одной стороны — «клиент» (покупатель с семьёй/друзьями и ограничениями),
с другой — «кассир» (голосовой ассистент). Оба агента реализованы через
большую языковую модель (LLM) с разными системными промптами и разными
ролями. Меню ресторана подключается как база знаний для семантического
поиска (RAG на ChromaDB + эмбеддинги). Профили посетителей и их групп
генерируются стохастически по заданным распределениям (условно «REG» —
rule/statistics-based generation профиля, а не retrieval, но в документации
проекта это явно названо рядом с RAG как второй источник контекста для
диалога).

Развёрнуто: проект сочетает (а) прикладную задачу — персонализированный
голосовой заказ без экрана на английском, (б) исследовательскую/методологическую
задачу — массовую синтетическую генерацию диалогов, их сохранение в
структурированном виде (JSON), автоматическую валидацию по аллергенам и
калориям, агрегирование статистики и сравнение режимов «кассир с RAG по меню»
и «кассир без RAG», чтобы оценивать влияние доступа к меню на качество
ответов. Таким образом, это не только «чат-бот», а воспроизводимый конвейер
эксперимента: профиль → диалог → артефакты → метрики.

--------------------------------------------------------------------------------
2. ЦЕЛИ ПРОЕКТА (ЧТО МЫ ХОТИМ ДОСТИЧЬ)
--------------------------------------------------------------------------------

2.1. Прикладные цели
  - Показать работоспособную архитектуру голосового ассистента fast-food:
    меню как база знаний, учёт аллергенов и калорийности, учёт состава
    группы (сам клиент + дети + друзья/спутники с отдельными ограничениями).
  - Обеспечить сценарий, в котором клиент заказывает «за всех», а кассир
    ведёт себя вежливо и последовательно, опираясь на RAG при наличии.

2.2. Исследовательские и дипломные цели (в терминах научного руководителя)
  - Построить двухсторонний диалоговый сервис, в котором для каждой стороны
    задана своя роль, свой промпт и (в случае кассира) свой доступ к
    внешней базе знаний (меню).
  - Получить возможность масштабировать генерацию диалогов (в документации
    проекта указан ориентир порядка тысяч диалогов; в идеале научрук
    формулирует порядок 10 000 для статистики) и затем анализировать ошибки
    LLM не «вручную по одному чату», а через структурированные логи,
    флаги валидации и сводки.
  - Связать распределения свойств людей (аллергии, лактаза, веганство и т.д.)
    с содержимым меню (поле аллергенов у позиций), чтобы автоматически
    ловить конфликты «заказали — но нарушили ограничение участника группы».

--------------------------------------------------------------------------------
3. КАК ФОРМУЛИРОВАЛ ЗАДАЧУ НАУЧНЫЙ РУКОВОДИТЕЛЬ И КАК ЭТО СООТНОСИТСЯ
    С РЕПОЗИТОРИЕМ (НАУЧНАЯ ЧАСТЬ «ПО ЕГО СЛОВАМ»)
--------------------------------------------------------------------------------

Ниже — не дословная стенограмма, а содержательная фиксация требований из
запроса студента, с явной привязкой: «что имелось в виду» → «что уже есть в
коде/данных» → «чего ещё нет / что слабее всего закрыто».

3.1. «Простая модель RAG / база знаний: текстовый файл с описанием покупателя»
  Идея научрука: сначала статистика и простая база знаний о человеке, потом
  промпт к LLM, чтобы эмулировать конкретного собеседника.

  В репозитории:
  - База знаний по меню — это не один текстовый файл, а mcd.json (42 позиции)
    плюс векторный индекс Chroma (chroma_db/), строится скриптом
    scripts/load_chroma.py; для эмбеддингов используется
    sentence-transformers/all-MiniLM-L6-v2 (384 измерения), метаданные включают
    аллергены, фильтры по калориям и аллергенам поддерживаются в поиске.
  - «Описание покупателя» формируется из структурированного профиля
    (генератор в src/mcd_voice/profile/generator.py): пол, возраст, психотип,
    язык, целевая калорийность, булевы ограничения (молоко, сахар, говядина,
    веган, рыба, орехи, яйца, глютен), спутники (дети с возрастом и своими
    вероятностями ограничений, друзья). Есть функция текстового описания
    профиля для подстановки в промпт (generate_text_description и связанные
    шаги в конвейере).
  - То есть: идея «сначала модель/описание клиента, потом LLM играет роль»
    реализована как JSON-профиль + текстовое описание, а не как один плоский
    .txt «через пробелы» — это расхождение с буквальной формулировкой, но
    содержательно тот же слой «база признаков посетителя».

3.2. «Эмулируем обе стороны: клиент и продавец; потом любую сторону заменяем
     на реальную — тренируем бота»
  Идея: симуляция двусторонняя → массовые данные → затем human-in-the-loop
  или замена одного агента реальным человеком.

  В репозитории:
  - Реализованы два класса агентов (src/mcd_voice/llm/agent.py): ClientAgent и
    CashierAgent, разные системные промпты (src/mcd_voice/llm/prompts.py).
  - Конвейер диалога (src/mcd_voice/dialog/pipeline.py) поочерёдно вызывает
    модель для кассира и клиента, ведёт историю, парсит заказ по персонам.
  - Замена «одной стороны на человека» на уровне кода как отдельного режима
    UI/телефонии в этом репозитории не делается: это следующий этап интеграции
    (голосовой ввод, стриминг, endpoint). Но архитектурно роли разделены,
    что соответствует требованию «две схемы».

3.3. «Сгенерировать 10 000 диалогов, искать косяки, локализовать ошибки,
     два промпта и два пайплайна с двумя RAG»
  Идея: массовая генерация → error mining → улучшение промптов; отдельные
  пайплайны для сторон.

  В репозитории:
  - Массовая генерация: scripts/generate_dataset.py с параметром --num_dialogs
    (пример в README: 100 диалогов; масштаб до 10 000 — это вопрос бюджета API
    и времени, а не ограничения программы).
  - Каждый диалог сохраняется в dialog_NNNN.json (профиль, история, order_state,
    флаги валидации); aggregate_stats() формирует summary.json.
  - Сравнение режимов: scripts/compare_rag.py для пары каталогов RAG vs
    non-RAG (таблица, rag_comparison.json, при наличии matplotlib — график).
  - «Два разных RAG» в буквальном смысле (отдельная векторная база для клиента
    и для кассира) сейчас не требуется предметной областью: клиент «не знает
    меню» и задаётся профилем; кассир — единственный, кто ходит в Chroma.
    Научная постановка «две модели принятия решения поверх общего сервиса»
    закрывается парой агентов и разными промптами; второй RAG для клиента
    можно добавить, если захотим моделировать «клиент с телефоном и подсказкой
    из приложения».

3.4. «Граф случайных решений: монетки — мужчина/женщина, один/семья, аллергии,
     проценты лактозы и т.д.; потом оборачиваем в LLM; клиент не знает меню
     и может просить невозможного»
  Идея: быстрый procedural generation + LLM для естественной речи.

  В репозитории:
  - ProfileGenerator как раз задаёт распределения (в PROJECT_CONTEXT.md
    перечислены ориентиры: пол, возрастные группы, психотипы, язык, калории,
    доли ограничений по питанию, дети/друзья по ВЦИОМ и т.д.).
  - Кассир использует distance threshold (0.60): если семантически далеко,
    отвечает, что позиции нет в меню — это частично покрывает «невозможные»
    запросы.
  - Полная модель «клиент намеренно просит абсурд» vs «кассир корректно
    отказывает» зависит от качества LLM и промптов; валидация после диалога
    ловит нарушения аллергенов и экстремальные калории, но не все виды
    логических ошибок.

3.5. «История клиента (хэш)»
  В репозитории на момент описания: в явном виде долговременная «история
  клиента с хэшем идентичности» как отдельная таблица/ключ не описана в
  PROJECT_CONTEXT.md как готовый модуль. Это прямой пункт доработки под
  научрука: связать повторные визиты, персонализацию, A/B тесты промптов на
  одной и той же «личности».

4. ЧТО КОНКРЕТНО РЕАЛИЗОВАНО В КОДЕ (ПО СЛОЯМ)
--------------------------------------------------------------------------------

4.1. Пакет src/mcd_voice/menu/ — RAG по меню
  - Парсинг меню из JSON, подготовка текста для эмбеддингов (имя, описание,
    ингредиенты, тег).
  - Загрузка в Chroma, коллекция menu, косинусная метрика HNSW.
  - search_menu с фильтрами по аллергенам (через not_contains) и калориям.
  - Тесты на поиск и фильтры: tests/test_search.py и др.

4.2. Пакет src/mcd_voice/profile/ — генерация посетителя и группы
  - generate_profile(), get_allergen_blacklist(), get_group_allergen_blacklist(),
    generate_text_description().
  - Учёт спутников с отдельными ограничениями (важно для семейных сценариев).

4.3. Пакет src/mcd_voice/llm/ — агенты
  - ClientAgent: знает профиль и группу, формулирует запросы от имени группы.
  - CashierAgent: адаптация к психотипу, поочерёдное уточнение, вызов RAG,
    fallback «нет в меню» по порогу расстояния.

4.4. Пакет src/mcd_voice/dialog/ — оркестрация и постанализ
  - DialogPipeline, simulate_dialog (публичный API пакета).
  - order_state: несколько персон (self, spouse по эвристикам текста, дети,
    друзья), парсинг количеств («3 Big Mac»).
  - Валидация: allergen_violation по персонам, calorie_warning относительно
    calApprValue, empty_order, агрегаты total_items, total_energy, turns.
  - Сохранение/загрузка JSON, aggregate_stats для summary.json.
  - Вспомогательные модули: каталог калорий, маппинг флагов ограничений на
    токены аллергенов меню.

4.5. scripts/ — сценарии запуска
  - load_chroma.py — индексация меню.
  - generate_dataset.py — массовая генерация (флаги RAG / no RAG).
  - compare_rag.py — сравнительный анализ двух наборов диалогов.
  - dialog_demo.py, profile_demo.py, agents_demo.py, test_menu_search.py,
    menu_search_demo.py — демонстрации и ручные проверки.

4.6. tests/ — автоматические тесты
  - В README и PROJECT_CONTEXT.md указано 55 pytest-тестов без необходимости
    в API-ключе для основной части проверок (RAG, профили, парсинг, валидация,
    save/load). Это аргумент воспроизводимости для диплома.

7. ЧТО УЖЕ ПОЛУЧИЛИ (РЕЗУЛЬТАТ НА ТЕКУЩИЙ МОМЕНТ)
--------------------------------------------------------------------------------

- Законченный сквозной прототип: профиль → двусторонний LLM-диалог →
  структурированный заказ по персонам → автоматические проверки → JSON.
- Воспроизводимый RAG-слой с фильтрацией по аллергенам и калориям, порог
  уверенности для «нет в меню».
- Возможность пакетно генерировать наборы диалогов и сравнивать качество
  RAG vs non-RAG на агрегатах, а не «на глаз».
- Тестовое покрытие ключевой логики без оплаты токенов на каждый прогон.

Это можно позиционировать в дипломе как: «реализована экспериментальная
платформа для исследования ошибок LLM в двухролевом сценарии fast-food с
учётом меню и ограничений здоровья».

--------------------------------------------------------------------------------
8. НАСКОЛЬКО ТЕКУЩЕЕ СОСТОЯНИЕ СООТВЕТСТВУЕТ «МАКСИМАЛЬНОМУ» ЗАДАНИЮ
    НАУЧНОГО РУКОВОДИТЕЛЯ
--------------------------------------------------------------------------------

Сильные совпадения:
  - Двусторонняя эмуляция (клиент + кассир) — да.
  - База знаний по меню + аллергены — да (RAG + метаданные).
  - Стохастические «монетки» для образа посетителя и группы — да (генератор).
  - Массовая генерация и последующий анализ через JSON/summary — да (скрипты).
  - Разделение промптов и поведения сторон — да.

Частичные совпадения / упрощения относительно устной постановки:
  - «Простой текстовый файл покупателя через пробелы» заменён структурой JSON
    и типизированными полями — это скорее плюс для науки (меньше шума, проще
    валидировать), но нужно явно написать в тексте диплома, почему так.
  - Масштаб 10 000 диалогов как выполненный эксперимент в репозитории не
    фиксирован (это запуск и ресурсы); инфраструктура к этому готова.
  - «Две базы RAG» для обеих сторон — не реализовано; вместо этого одна база
    меню для кассира и профиль для клиента.
  - «Локализация любых ошибок LLM» — реализованы жёсткие проверки по аллергенам
    и калориям и порог RAG; тонкий семантический анализ (галлюцинации цены,
    неверный состав, токсичность, нарушение политики бренда) потребует
    дополнительных классификаторов или вторичной модели-судьи.

9. ЧТО ЕЩЁ НУЖНО ДЕЛАТЬ (ROADMAP ПОД ДИПЛОМ И ПО ЗАМЕЧАНИЯМ НАУЧРУКА)
--------------------------------------------------------------------------------

9.1. Обязательный минимум для «закрытия» научной части в духе руководителя
  A) Прогнать крупный корпус диалогов (целевой порядок: 3 000–10 000) в двух
     режимах (RAG / non-RAG) на одной и той же версии кода и зафиксировать
     гиперпараметры (модель, температура, max_turns, порог distance).
  B) Формализовать методику поиска ошибок: не только aggregate_stats, но
     список типов ошибок (аллерген, калории, «нет в меню» но заказали,
     дублирование позиций, несогласованность персон, обрыв диалога) и
     скрипт выборки примеров по каждому типу.
  C) Описать в тексте диплома связку «распределение ограничений в популяции
     симулированных клиентов» ↔ «аллергены в меню» ↔ «доля диалогов с
     нарушением / предупреждением» — это прямой ответ на идею процентов
     лактозы и т.п.

9.2. Усиление соответствия формулировкам про «историю клиента (хэш)»
  - Ввести стабильный идентификатор профиля (hash от нормализованного JSON
    профиля или от seed + версии генератора).
  - Сохранять серию диалогов для одного id (имитация повторных визитов).
  - Показать, как меняется поведение при подмешивании «памяти» в промпт
    кассира (осторожно с ПДн — в дипломе лучше синтетика без реальных людей).

9.3. Продвинутые направления (если хватит времени)
  - Второй лёгкий RAG для клиента (например, «видит рекламу/приложение») —
    отдельная ветка эксперимента.
  - Классификатор ошибок на базе маленькой модели или эвристик поверх логов.
  - Голосовой контур (STT/TTS) как демонстрация, не ядро науки.
  - Подключение более богатого меню и проверка устойчивости метрик.

9.4. Оформление диплома
  - Чётко выделить: «прикладной модуль», «методология симуляции»,
    «эксперимент», «этика и ограничения» (синтетические персонажи; избегать
    дискриминационных шаблонов в примерах профилей — в коде используются
    нейтральные поля sex/age/psycho, без этнических ярлыков; если научрук
    приводил бытовые примеры, в реализации проекта следует придерживаться
    этичного минимума и документировать это).
  - Воспроизводимость: версии зависимостей, команда запуска, что именно
    лежит в chroma_db.

--------------------------------------------------------------------------------
10. КРАТКОЕ РЕЗЮМЕ ДЛЯ ВСТАВКИ В ДИПЛОМ (АБЗАЦ)
--------------------------------------------------------------------------------

В работе разработана и реализована платформа двусторонней LLM-симуляции
диалога заказа в ресторане быстрого питания с векторным поиском по меню
(RAG), стохастической генерацией профилей посетителей и их групп,
многоперсональным учётом заказа и пост-валидацией по аллергенам и калориям.
Платформа позволяет массово генерировать диалоги, сохранять их в JSON,
агрегировать статистику и сравнивать режимы с доступом к меню и без него.
Научная ценность — в воспроизводимом конвейере генерации и анализа ошибок
языковых моделей в предметной области с ограничениями здоровья; дальнейшее
развитие связано с наращиванием корпуса диалогов, типологией ошибок,
учётом «истории клиента» и расширением метрик качества.

11. ТАБЛИЦА СООТВЕТСТВИЯ: СЛОВА НАУЧНОГО РУКОВОДИТЕЛЯ → РЕПОЗИТОРИЙ
--------------------------------------------------------------------------------

Требование / идея научрука          | Статус в проекте
-------------------------------------|-----------------------------------------------
База знаний по блюдам и аллергенам   | RAG: mcd.json + Chroma + фильтры allergens
Простой текстовый профиль клиента    | JSON-профиль + текстовое описание (не .txt)
Стохастический «граф монеток»        | ProfileGenerator, распределения в generator.py
Две стороны диалога (клиент/кассир)   | ClientAgent + CashierAgent, DialogPipeline
Два пайплайна / два промпта          | Разные system prompts; один общий конвейер
Массовая генерация (тысячи диалогов) | generate_dataset.py, параметр num_dialogs
Поиск ошибок LLM по корпусу          | Частично: валидация + summary; нужна типология
Проценты аллергий vs меню            | Данные: флаги профиля + allergens в меню; нужен отчёт
Клиент не знает меню                 | Заложено ролью клиента; кассир — RAG
История клиента (хэш)                | План доработки, в коде не как отдельный модуль
Замена одной стороны на человека     | Архитектурно возможно; UI/голос — вне репозитория

12. НАУЧНАЯ НОВИЗНА И ВКЛАД (КАК ФОРМУЛИРОВАТЬ В ГЛАВЕ ДИПЛОМНОЙ)
--------------------------------------------------------------------------------

Новизна не в том, что «впервые в мире сделан чат-бот Макдональдса», а в связке
методов для воспроизводимого исследования:

  (1) Двухагентная LLM-симуляция с явным разделением ролей и доступа к знаниям
      (асимметрия: только кассир видит retrieval по меню).

  (2) Процедурная генерация популяции клиентов с ограничениями на уровне группы
      (не только «один человек», но дети/друзья со своими blacklist аллергенов).

  (3) Пост-диалоговая формальная проверка согласованности заказа с медицинско-
      пищевыми ограничениями участников (аллергены из карточек блюд) и энергией.

  (4) Конвейер эксперимента RAG vs non-RAG на одной и той же генерации профилей
      и сценариях, с автоматическим сводным отчётом.

Вклад практический: готовый каркас для тренировки и оценки будущего голосового
ассистента без ручной разметки каждого диалога — симуляция даёт объём, формальные
проверки дают нижнюю границу качества по безопасности меню.

13. МЕТОДОЛОГИЯ ЭКСПЕРИМЕНТА (ПОШАГОВО, ДЛЯ РАЗДЕЛА «МЕТОДЫ»)
--------------------------------------------------------------------------------

Шаг 1. Подготовка среды: установка зависимостей, переменные API (OpenAI или
       Ollama), при необходимости .env.

Шаг 2. Индексация меню: python scripts/load_chroma.py → chroma_db/.

Шаг 3. Верификация кода без платных вызовов: pytest tests/ -v.

Шаг 4. Пилот: небольшой num_dialogs (10–50) для отладки промптов и лимитов
       max_turns.

Шаг 5. Основной эксперимент: два каталога выходов — с RAG и --no_rag, одинаковый
       num_dialogs, фиксация model и seed (если добавите seed в скрипт — зафиксировать
       в дипломе).

Шаг 6. Сравнение: python scripts/compare_rag.py → численные и графические
       итоги при наличии matplotlib.

Шаг 7. Интерпретация: не только «кто лучше», но какие failure modes чаще
       (требует доработки скриптов выборки по флагам и по ключевым словам в
       истории).

14. ОГРАНИЧЕНИЯ И ЧЕСТНОСТЬ ПЕРЕД КОМИССИЕЙ
--------------------------------------------------------------------------------

- Качество симуляции ограничено качеством LLM и стоимостью/скоростью API.
- Меню и аллергены в mcd.json — учебный/упрощённый датасет; клинические решения
  по питанию на их основе не принимаются.
- Этника: персонажи синтетические; при расширении профилей нельзя вводить
  чувствительные категории как «карикатуры» для юмора модели — только
  обоснованные демографические и поведенческие признаки, согласованные с
  научруком.
