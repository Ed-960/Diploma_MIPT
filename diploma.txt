АННОТАЦИЯ
================================================================================

Объём: [__] стр., [__] рис., [__] табл., [__] ист.

Работа посвящена разработке программного комплекса для моделирования и оценки
качества диалогов голосового ассистента, принимающего заказ в ресторане быстрого
обслуживания (Quick Service Restaurant, QSR). Предложен интегрированный конвейер,
сочетающий retrieval-augmented generation (RAG) по структурированному меню в двух
вариантах: векторный (dense retrieval в векторной базе Chroma) и графовый
(расширение по взвешенному графу согласованности позиций меню), стохастическую
генерацию профилей клиентов (rule-enhanced generation, REG), двустороннюю
симуляцию диалога с участием двух LLM-агентов (роли клиента и кассира) и
формальную автоматическую верификацию итогового заказа относительно ограничений
профиля и справочника блюд.

Описаны предметная область, обзор литературы, требования к системе, архитектура
и реализация прототипа на языке Python. Приведена программа экспериментального
сравнения режимов retrieval; количественные результаты экспериментов на этапе
подготовки данного текста не фиксируются и планируются к включению после
проведения измерений.

Ключевые слова: большие языковые модели; диалоговые системы; RAG; графовый
retrieval; синтетические диалоги; верификация заказа; ресторан быстрого
обслуживания.

Перечень сокращений и обозначений:
  LLM — Large Language Model (большая языковая модель).
  QSR — Quick Service Restaurant (ресторан быстрого обслуживания).
  RAG — Retrieval-Augmented Generation (генерация с подстановкой извлечённого контекста).
  REG — Rule-Enhanced Generation (в работе: управляемая стохастическая генерация профиля).
  ASR — Automatic Speech Recognition (автоматическое распознавание речи).
  TTS — Text-to-Speech (синтез речи).
  API — программный интерфейс приложения.
  JSON — текстовый формат обмена данными.

ВВЕДЕНИЕ
================================================================================

Рестораны быстрого питания (QSR) все шире используют цифровые каналы взаимодействия
с клиентом: мобильные приложения, киоски самообслуживания, интеграцию с агрегаторами
доставки. Отдельным направлением является голосовой приём заказа, в том числе в
сценарии «окно выдачи» (drive-through), когда визуальная обратная связь ограничена
или отсутствует. Для таких сценариев естественно рассматривать разговорного
ассистента на базе больших языковых моделей (Large Language Models, LLM), однако
прямое применение генеративной модели без опоры на контролируемые данные опасно:
модель может предложить несуществующее блюдо, неверно описать состав, проигнорировать
аллерген или допустить логическую несогласованность заказа с профилем клиента.

Стандартным инженерным ответом на риск «галлюцинаций» относительно фактов является
подход retrieval-augmented generation (RAG): в контекст генерации подставляются
фрагменты из внешней базы знаний — в задаче меню это записи о блюдах, ингредиентах,
аллергенах и нутриентах. Одного RAG недостаточно: даже при корректном извлечении
ассистент должен согласовать рекомендации с ограничениями пользователя (в том числе
пищевыми), учитывать состав группы (дети, сопровождающие) и поддерживать структуру
заказа на протяжении длинного диалога. Кроме того, для настройки и регрессионного
тестирования требуется большой объём диалоговых траекторий, тогда как сбор реальных
логов с полными профилями затруднён требованиями конфиденциальности.

В связи с этим в работе рассматривается программный конвейер, который:
(1) формирует синтетические, но статистически управляемые профили клиентов (REG);
(2) симулирует двусторонний диалог «клиент — кассир» силами двух LLM-агентов;
(3) использует два режима извлечения информации о меню — векторный и графовый;
(4) после диалога выполняет автоматическую верификацию заказа по формальным правилам.

Объектом исследования является процесс автоматизированного приёма заказа в QSR
при использовании ассистента на базе LLM. Предметом исследования являются методы
и программная реализация генерации синтетических диалогов, retrieval по меню
и автоматической верификации корректности итогового заказа.

Целью работы является разработка программного конвейера генерации и верификации
диалогов для голосового ассистента заказа в QSR, обеспечивающего воспроизводимость
сценариев и автоматическое выявление критических ошибок.

Для достижения цели решаются следующие задачи:
  — провести обзор литературы и практик по голосовым ассистентам в сфере питания,
    RAG, генерации синтетических диалогов и оценке качества; сформулировать
    требования к системе;
  — разработать архитектуру конвейера и описать алгоритмы векторного и графового
    retrieval, генерации профилей, ведения диалога и валидации;
  — реализовать прототип на языке Python в виде библиотеки и исполняемых сценариев;
  — разработать программу экспериментального сравнения режимов retrieval и анализа
    ошибок (с последующим заполнением численных результатов после проведения
    экспериментов).

Методы исследования: анализ научных и прикладных источников; проектирование
программных модулей; стохастическое моделирование профилей; агентное моделирование
на LLM; rule-based верификация; план статистического анализа агрегированных метрик.

Научная новизна работы (на уровне инженерной постановки) заключается в согласованном
объединении управляемой генерации профилей, двух альтернативных стратегий retrieval
по меню, двусторонней LLM-симуляции и формальной верификации инвариантов заказа
в едином воспроизводимом контуре.

Практическая значимость: предложенный конвейер пригоден для масштабируемого
тестирования политики ассистента до пилотного внедрения, сравнения версий промптов
и конфигураций поиска, а также для накопления размеченного корпуса диалогов с
флагами ошибок.

Структура работы: глава 1 посвящена предметной области и обзору литературы; глава 2 —
постановке задачи и требованиям; глава 3 — архитектуре и алгоритмам; глава 4 —
реализации прототипа; глава 5 — программе экспериментальной апробации. В заключении
сформулированы итоги и направления развития.

ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ОБЗОР ЛИТЕРАТУРЫ
================================================================================

1.1. Голосовые интерфейсы и заказ еды в QSR: задачи и ограничения

Голосовой заказ в ресторане быстрого питания сочетает ограничения распознавания речи
с прикладными требованиями корректности содержания: пользователь формулирует запрос
естественным языком, возможны шум, акцент, неполные формулировки. Исследования,
опирающиеся на анализ отзывов и инцидентов в онлайн-заказах еды, показывают, что
проблемы качества и безопасности пищи остаются значимой темой, хотя напрямую они
не исчерпывают задачу голосового ассистента [1]. Обзорные работы по облачным
голосовым системам заказа для QSR подчёркивают потенциал NLP и облачной инфраструктуры,
но также указывают на вызовы конфиденциальности и интеграции [2]. Прототипы на базе
встраиваемых платформ (например, Raspberry Pi) демонстрируют цепочку ASR — правила —
Seq2Seq для нестандартных запросов [3]; веб-подходы с Web Speech API и NLP показывают
практическую осуществимость, но не гарантируют согласованности с актуальным меню
и медицинскими ограничениями без отдельного модуля знаний [4].

Вывод по направлению: типичные системы закрывают сценарий «голос — намерение —
оформление», но слабо связаны со структурированным меню, фильтрацией аллергенов
и масштабируемой верификацией на больших наборах диалогов — что и определяет место
предлагаемого конвейера.


1.2. Критерии качества ассистента в задаче оформления заказа

Для задач ориентированных диалоговых систем принято оценивать: корректность
понимания намерения, полноту заполнения слотов заказа, соответствие доменным
ограничениям, устойчивость к перефразированию и ошибкам ASR. В прикладных RAG-проектах
встречаются метрики качества извлечения (retrieval accuracy), а также классические
метрики совпадения с эталоном (BLEU, ROUGE, METEOR) при сравнении генераций [5, 6].
Для классификационных постановок (например, сопоставление номенклатур) приводятся
accuracy и анализ неоднозначности формулировок [7].

В контексте настоящей работы важны дополнительно бинарные и структурные показатели:
наличие аллергенного конфликта в итоговом заказе, факт галлюцинации (позиция вне
меню), неполнота заказа относительно профиля (например, дети без отдельных позиций).
Это сближает постановку с задачами верификации ограничений, а не только с оценкой
«естественности» текста.


1.3. RAG и учёт ограничений при работе с меню

RagU (Santa Clara University, 2024) описывает разговорного ассистента меню с веб-сбором
данных, SQLite и интеграцией ChatGPT; отмечается высокая точность извлечения на
контрольных запросах, а также учёт аллергенов через UI и инструкции промпта [5].
Работы по практической реализации RAG с открытыми LLM и ChromaDB обсуждают влияние
размера чанков и выбор метрик качества [6].

Ключевой вывод для проектирования: RAG снижает риск «свободного изобретения» состава,
но требует явной связки извлечённых позиций с фильтрацией по метаданным (аллергены,
энергетическая ценность) и последующей проверкой согласованности с пользовательским
профилем на уровне правил, а не только на уровне языковой модели.


1.4. Синтетические диалоги и двустороннее моделирование

Генерация синтетических данных для экспериментов с LLM широко используется; отдельные
работы формализуют влияние формата представления структурированных данных на точность
ответов модели [8]. Для диалогов распространён приём «двух агентов» на одной модели
с разными системными инструкциями, что позволяет получать разнообразие сценариев без
живых аннотаторов, но требует внешней валидации фактической корректности.

В рассматриваемом проекте эта валидация возложена на детерминированные проверки относительно
каталога меню и blacklist аллергенов, а также на накопление структурированного order_state.


1.5. Ошибки LLM и автоматическая проверка

Исследования по выявлению галлюцинаций рассматривают комбинации RAG, chain-of-thought
и ансамблирование моделей как способы повышения надёжности [9]. Для прикладной системы
заказа еды недостаточно общего «детектора галлюцинаций»: необходима проверка на уровне
объектов домена (позиции меню, аллергены по персонам, энергия порций).


1.6. Семантический поиск и векторные хранилища

Сравнительные работы по производительности Chroma, Qdrant и Faiss показывают зависимость
ускорения поиска и вставки от объёма данных и режима хранения (память против диска) [10].
Для небольшого каталога меню (в реализации — 42 позиции) критичнее качество эмбеддингов
и корректность метаданных, чем экстремальная масштабируемость индекса.


1.7. Выводы по главе 1

Проведённый обзор подтверждает актуальность связки «RAG по меню + персонализированный
диалог + формальная верификация». Литература по отдельным компонентам представлена
шире, чем интегрированные решения с управляемой генерацией профилей и сравнением
двух режимов retrieval в одном конвейере — что определяет направление собственной
разработки.

ГЛАВА 2. ПОСТАНОВКА ЗАДАЧИ И ФОРМИРОВАНИЕ ТРЕБОВАНИЙ К СИСТЕМЕ
================================================================================

2.1. Объект и предмет исследования

Исходя из методических рекомендаций, объект исследования относится к более широкой
области: процессы автоматизированного приёма заказов в QSR при применении LLM.

Предмет исследования сужается до конкретного набора методов и программных механизмов:
генерация синтетических сценариев (профили, диалоги), retrieval по структурированному
меню (векторный и графовый варианты) и автоматическая верификация соответствия
заказа ограничениям и каталогу.


2.2. Цель работы и задачи

Цель работы сформулирована во введении. Задачи детализируются следующим образом.

Задача 1 — аналитико-обзорная: систематизировать сведения по предметной области,
выявить пробелы и сформулировать требования.

Задача 2 — проектная: определить модульную архитектуру, модель данных меню и
профиля, интерфейсы между RAG, REG, LLM и модулем диалога.

Задача 3 — программная: реализовать прототип, прошедший регрессионные тесты в
репозитории проекта.

Задача 4 — методическая: разработать программу экспериментов по сравнению режимов
retrieval и анализу распределения типов ошибок.

Задача 5 — рекомендательная: сформулировать пути интеграции голосового контура
и промышленной эксплуатации.


2.3. Функциональные и нефункциональные требования

Функциональные требования:
  FR1 — генерация профиля клиента и сопровождающих с управляемыми распределениями;
  FR2 — ведение многоходового диалога на английском языке (в текущей реализации
        диалоговая симуляция текстовая; голосовой ввод-вывод рассматривается как
        следующий этап);
  FR3 — retrieval в режиме vector и в режиме graph;
  FR4 — поддержка группового заказа (несколько персон в order_state);
  FR5 — автоматическая валидация: аллергены, калорийные предупреждения, пустой заказ,
        неполнота, галлюцинации по позициям;
  FR6 — сохранение диалогов и агрегированной статистики в JSON.

Нефункциональные требования:
  NFR1 — воспроизводимость (фиксация seed, параметров запуска);
  NFR2 — разделение библиотечного кода (`src/mcd_voice`) и сценариев (`scripts`);
  NFR3 — наблюдаемость (опциональные трассы RAG/LLM);
  NFR4 — возможность пакетной генерации большого числа диалогов (порядка тысяч);
  NFR5 — допустимость использования облачного OpenAI API либо локального OpenAI-
         совместимого провайдера (Ollama).


2.4. Обоснование новизны и практической значимости

Отличие разрабатываемого конвейера от ближайших известных аналогов (по уровню
инженерной постановки) заключается в одновременном наличии:
  — двух альтернативных механизмов retrieval по одному и тому же каталогу меню;
  — стохастического профиля с формальным decision graph;
  — двусторонней LLM-симуляции с последующей rule-based верификацией.

Преимущества: измеримость ошибок на больших наборах, возможность регрессионного
анализа при изменении промптов и порогов RAG.

Выгода для потребителя (конечного пользователя продукта): снижение риска ошибок
заказа, связанных с аллергенами и несоответствием меню; для разработчика — снижение
стоимости ручного тестирования.


2.5. Выводы по главе 2

Сформулированы цель, задачи, объект и предмет, требования к программному конвейеру
и обоснование практической полезности, что задаёт основу для архитектурного
описания в главе 3.

ГЛАВА 3. АРХИТЕКТУРА И АЛГОРИТМЫ ПРОГРАММНОГО КОНВЕЙЕРА

3.1. Общая схема и принципы декомпозиции

Прототип организован в виде пакета `mcd_voice` со слоями:
menu (данные и retrieval), profile (REG), llm (агенты), dialog (оркестрация,
парсинг, валидация). Исполняемые сценарии вынесены в `scripts`, что отделяет
исследовательские и производственные запуски от библиотечного API.

Такая декомпозиция соответствует принципу единственной ответственности: замена
модели эмбеддингов, провайдера LLM или стратегии retrieval не должна требовать
переписывания всего конвейера.

На рисунке Г.1 (приложение Г) приведена обобщённая схема потока данных: профиль
и меню подаются в диалоговый контур; кассир использует один из режимов retrieval;
по завершении диалога формируется структурированный заказ и вычисляются флаги
валидации, которые агрегируются при массовой генерации.


3.2. Модель данных меню и подготовка к RAG

Исходный каталог хранится в `mcd.json` и содержит 42 позиции. Для каждой записи
формируется текст документа для эмбеддинга из полей name, description, ingredients,
tag (пустые поля опускаются). В метаданные для Chroma включаются наименование,
категория, список аллергенов (с маппингом пустого набора на служебный маркер для
фильтрации), числовые нутриенты, включая energy, сахара, натрий и др.

Семантический поиск использует косинусную метрику в пространстве эмбеддингов модели
`sentence-transformers/all-MiniLM-L6-v2` (размерность 384). Для фильтрации аллергенов
применяется построение условия `where` по blacklist из профиля клиента.

В коде используется извлечение по полному каталогу позиций (top_k соответствует числу
позиций меню), после чего применяются пороговые правила по «дистанции» совпадения
и ограничение числа строк, попадающих в системный промпт кассира — это позволяет
контролировать размер контекста без потери возможности фильтрации по метаданным.


3.3. Векторный режим retrieval

Векторный режим реализован через Chroma persistent client и функцию поиска с
ограничением по расстоянию: задаётся порог «уверенного» попадания (`RAG_DISTANCE_THRESHOLD`,
по умолчанию 0.60 в косинусной метрике) и мягкий порог для частичного включения строк
меню в промпт кассира. Полный каталог позволяет извлекать top-k по всей коллекции с
последующей фильтрацией и отсечением по порогу,
что согласуется с требованием не допускать выдуманных позиций при плохом совпадении
запроса с меню.


3.4. Графовый режим retrieval

Графовый режим не требует отдельной graph database: строится взвешенный граф на
множестве позиций меню. Узлы соответствуют блюдам; рёбра возникают при ненулевой
схожести по категории, тегу, пересечению множеств аллергенов и текстовых токенов
(ингредиенты и описания). Вес рёбра суммирует вклад совпадений.

Поиск выполняется в два этапа: лексически-подобный подсчёт релевантности запроса
к узлам (seeds), затем распространение оценки на соседей с ограничением числа
расширений и ранжирование кандидатов; позиции, несовместимые с blacklist аллергенов,
исключаются. Для трассировки в метаданных фиксируется режим `graph` и служебные
поля (число кандидатов, seeds).


3.5. Генерация профилей (REG) и граф решений

Профиль включает демографию (пол, возраст), психотип (для стиля диалога), язык,
целевую ориентировочную калорийность, булевы ограничения на ингредиенты/диету,
а также параметры группы: число детей и друзей, массив companions с возрастом
и индивидуальными ограничениями. Распределения параметров задаются на основе
статистических ориентиров (Росстат, ВЦИОМ, данные о распространённости непереносимости
и аллергий и т.п., как зафиксировано в комментариях реализации).

В текущей версии генератора язык диалога для симуляции зафиксирован на английском:
в коде задано взвешенное распределение `EN: 1.0`, `RU: 0.0` (изменение весов позволяет
включить русскоязычные сценарии без переработки остального контура — см. приложение Ж).

Таблица 3.1 — пример ключевых маргинальных вероятностей (фрагмент, по коду генератора).

+------------------------+------------------------------+
| Параметр               | Значение / распределение     |
+------------------------+------------------------------+
| Пол male / female      | 0.46 / 0.54                  |
| Возрастная группа      | 18–30: 0.25; 31–55: 0.50;    |
|                        | 56–80: 0.25                  |
| Психотип regular       | 0.30                         |
| Психотип friendly      | 0.20                         |
| Психотип impatient     | 0.20                         |
| Психотип polite_and_   | 0.15                         |
| respectful             |                              |
| Психотип indecisive    | 0.15                         |
| Число детей 0/1/2/3    | 0.60 / 0.20 / 0.12 / 0.08    |
| Число друзей 0/1/2     | 0.83 / 0.12 / 0.05           |
| Маргинали noMilk (взр.)| 0.61 (логика веган-ветки     |
|                        | уточняет совместность)       |
+------------------------+------------------------------+

Отображение ограничений профиля на токены аллергенов в меню реализовано явным словарём
соответствия, например: noMilk → «Milk», noGluten → «Cereal containing gluten».

Для документирования и проверки эквивалентности реализации описан формальный
decision graph; экспорт в формат Mermaid поддерживается скриптом визуализации.
Альтернативно доступна генерация через явный обход графа — для валидации того,
что код соответствует заявленной структуре ветвлений.


3.6. LLM-агенты и диалоговый конвейер

Агент клиента и агент кассира реализованы как вызовы OpenAI-compatible Chat Completions
с разными системными промптами. Кассир получает доступ к результатам retrieval
(в зависимости от режима vector или graph), профилю и состоянию заказа; клиент
ориентируется на профиль и историю. История диалога передаётся целиком на каждом
ходу — это повышает согласованность, но увеличивает объём токенов и стоимость вызовов.

Конвейер `DialogPipeline` выполняет приветствие кассира (сценарий drive-through),
затем цикл «клиент — кассир», обновляя order_state на основе реплик клиента согласно
эвристикам парсинга названий блюд, количеств и адресации к персонам («for my child»,
«for everyone» и т.д.). Реализованы детекторы зацикливания и прерывания по лимиту
ходов.

На уровне реализованных алгоритмов используются:
  (а) регулярные выражения для распознавания завершения диалога;
  (б) жадный поиск вхождений названий блюд с локальным анализом префикса для извлечения
      количества (ограничение диапазона 1..20);
  (в) fallback-словарь алиасов (fries, nuggets, black coffee и т.п.) для неполных
      естественных формулировок;
  (г) rule-based назначение позиции конкретной персоне по паттернам («for my wife»,
      «for the youngest» и др.);
  (д) детектор зацикливания кассира по окну последних реплик.
Указанные алгоритмы имеют линейную по длине реплики сложность и хорошо подходят
для интерактивного конвейера при ограниченном размере меню.


3.7. Верификация заказа

После завершения диалога вычисляется набор флагов: нарушения аллергенов по каждой
персоне, предупреждение о калорийности относительно целевого ориентира, пустой заказ,
неполнота при наличии детей без позиций, признак наличия позиций вне каталога
(галлюцинации на уровне извлечённого заказа). Дополнительно формируется эвристическая
локализация ошибок по репликам для последующего анализа.

Таким образом, верификация опирается на структурированные данные меню и профиля
и не требует ручной разметки каждого шага диалога.


3.8. Выводы по главе 3

Описана архитектура и основные алгоритмы, соответствующие реализованному коду.
Сравнение режимов vector и graph retrieval и оценка влияния на ошибки диалога
выносится в экспериментальную главу.

ГЛАВА 4. РЕАЛИЗАЦИЯ ПРОТОТИПА
================================================================================

4.1. Структура репозитория и технологический стек

Проект использует Python 3.10+, пакетное оформление через `pyproject.toml`, зависимости
включают chromadb, sentence-transformers, openai и др. Тесты выполняются через pytest;
на момент подготовки текста в каталоге `tests/` собрано 92 автоматических теста
(`pytest --collect-only`), что обеспечивает регрессионный контроль модулей поиска,
профилей, пайплайна диалога, конфигурации LLM и сохранения артефактов.

Каталог `src/mcd_voice` содержит прикладной код; `scripts` — точки входа: загрузка
меню в Chroma, демонстрации, массовая генерация датасетов, сравнение наборов,
визуализация графов профиля и меню.


4.2. Реализация модулей

Модуль `menu` обеспечивает парсинг JSON-меню, построение текстов для эмбеддингов,
загрузку коллекции Chroma и функции поиска с фильтрами, а также графовый retrieval.

Модуль `profile` реализует класс `ProfileGenerator` и вспомогательные функции
чёрного списка аллергенов для клиента и группы.

Модуль `llm` содержит `ClientAgent`, `CashierAgent`, конфигурацию провайдера через
переменные окружения (`API_PROVIDER`, `OPENAI_API_KEY`, `OLLAMA_URL`, `API_MODEL` и др.).

Модуль `dialog` содержит `DialogPipeline`, функции симуляции, валидации, сохранения
диалогов и агрегирования статистики.


4.3. Сценарии запуска и воспроизводимость

Типичная последовательность для разработчика: создание виртуального окружения,
установка пакета в режиме editable, запуск `load_chroma.py` для построения индекса,
затем `generate_dataset.py` с параметрами числа диалогов, режима RAG (`--rag_mode vector|graph`)
или отключения RAG (`--no_rag`), при необходимости — файл профилей и seed для
повторяемой выборки.

Для массовой генерации реализована ранняя проверка доступности учётных данных LLM,
чтобы отказ конфигурации не приводил к многократному повторению одной и той же ошибки
в пуле потоков.


4.4. Ограничения прототипа

Основной контур симуляции — текстовый; полноценная интеграция ASR/TTS и оценка
латентности в полевых акустических условиях выходит за рамки текущей реализации
и рассматривается как отдельный этап. Интеграция с промышленными POS/CRM не реализована.
Экспериментальные отчёты с итоговыми таблицами метрик подлежат включению после
проведения измерений.


4.5. Выводы по главе 4

Прототип соответствует описанной архитектуре и пригоден для запуска экспериментальной
серии по сравнению режимов retrieval и анализу ошибок валидации.


================================================================================
ГЛАВА 5. ПРОГРАММА ЭКСПЕРИМЕНТАЛЬНОЙ АПРОБАЦИИ
================================================================================

5.1. Цель и гипотезы

Цель апробации — количественно сопоставить влияние режимов retrieval (отсутствие RAG,
векторный RAG, графовый RAG) на метрики ошибок, извлекаемые из валидации, при фиксированных
или парно сопоставимых профилях клиентов.

Рабочие гипотезы:
 H1 — векторный RAG обеспечивает более высокую релевантность подстановки меню на
      семантически неоднозначных запросах по сравнению с графовым режимом;
 H2 — графовый режим при заданных порогах показывает иной компромисс между релевантностью
      и соблюдением аллергенных ограничений;
 H3 — отсутствие retrieval увеличивает долю фактических несоответствий каталогу или
      ограничениям по сравнению с режимами с RAG.

Указанные гипотезы подлежат проверке после генерации достаточного объёма диалогов и
агрегирования `summary.json`.


5.2. Метрики и дизайн эксперимента

Первичные метрики берутся из валидации: доли диалогов с `allergen_violation`,
наличием «галлюцинированных» позиций, `incomplete_order`, а также распределение
`total_items`, `total_energy`, `turns`. Дополнительно анализируются трассы RAG,
если включён сбор.

Планируется парный дизайн по seed профилей: для одного и того же набора профилей
генерируются наборы диалогов под разными режимами retrieval, чтобы снизить дисперсию
от выбора клиентских сценариев.


5.3. Порядок проведения и статистическая обработка

После генерации наборов для каждого режима выполняется агрегирование через
`aggregate_stats()`, сравнение через `compare_rag.py` либо собственные скрипты
табулирования. Для долевых показателей планируется оценка доверительных интервалов
и проверка значимости различий (например, точный тест Фишера или критерий МакНемара
для парных сравнений, в зависимости от постановки).


5.4. Выводы по главе 5

На момент подготовки текста ВКР экспериментальные серии могут быть не завершены;
в финальной версии диплома в данную главу включаются таблицы результатов, графики
и интерпретация гипотез. Настоящий раздел фиксирует методику и воспроизводимый план
измерений без заполнения численных итогов.

ЗАКЛЮЧЕНИЕ

В работе рассмотрена задача построения программного конвейера для генерации и
верификации диалогов при моделировании голосового ассистента заказа в QSR на базе
LLM. Проведён обзор литературы, показавший, что отдельные компоненты (RAG, оценка
диалогов, детекция ошибок LLM) изучены шире, чем их согласованное сочетание с
управляемой генерацией профиля и формальной проверкой инвариантов заказа.

Сформулированы объект, предмет, цель и задачи; заданы требования к системе.
Описаны архитектура и алгоритмы векторного и графового retrieval, REG, диалогового
конвейера и валидации. Приведено описание реализованного прототипа и программа
экспериментальной апробации.

Практический результат: работоспособный программный комплекс в репозитории проекта,
позволяющий генерировать большие корпуса диалогов и автоматически вычислять метрики
ошибок на уровне заказа.

Направления дальнейшей работы: завершение экспериментальной серии и включение
количественных результатов; расширение голосового контура (ASR/TTS) и оценка
латентности; интеграция с учётными системами; дообучение или дистилляция моделей
при наличии накопленных данных.

Файл `dialogs/dialog_0001.json` (номер с ведущими нулями) формируется функцией

ПРИЛОЖЕНИЕ Е. ПСЕВДОКОД КЛЮЧЕВЫХ АЛГОРИТМОВ
================================================================================

Е.1. Векторный поиск меню (логика уровня агента)

  Вход: запрос пользователя, профиль (blacklist аллергенов), коллекция Chroma.
  1. Построить фильтр WHERE по аллергенам, исключаемым для клиента.
  2. Запросить top_k = |меню| ближайших векторов по косинусному расстоянию.
  3. Отфильтровать по порогу RAG_DISTANCE_THRESHOLD (по умолчанию 0.60).
  4. Часть строк с расстоянием между жёстким и мягким порогом — опционально в промпт
     кассира; при превышении мягкого порога строка не подмешивается.

Е.2. Графовый поиск (обобщённо)

  1. Построить граф позиций меню (кэшируется): узлы — имена блюд; рёбра — совпадения
     признаков из метаданных.
  2. Из запроса извлечь seed-узлы (лексика + совпадение с именами/описанием).
  3. Расширить ответ соседями в графе до заданного числа кандидатов, с учётом фильтра
     по аллергенам.

Е.3. Валидация итогового заказа

  Для каждой персоны: пересечение заказанных аллергенов с чёрным списком (self —
  из профиля; companion — из restrictions). Сверка имён позиций с каталогом меню,
  проверка калорий и полноты заказа для детей (см. docstring `validate_dialog` в коде).

Е.4. Извлечение количества и позиции из реплики клиента

  Вход: текст реплики, список menu_names.
  1. Нормализовать текст: lower-case, удалить trademark-символы, пунктуацию.
  2. Для каждого имени из menu_names проверить вхождение в текст.
  3. Если вхождение найдено, взять до 24 символов слева и извлечь число в конце префикса.
  4. Нормализовать количество в диапазон [1, 20], иначе qty=1.
  5. Если полных совпадений нет, применить alias-паттерны (fries/nuggets/coffee/...).
  Выход: список (item_name, quantity).

Е.5. Назначение позиции персоне (self/child/friend/spouse)

  1. Проверить реплику по набору regex-паттернов (_PERSON_PATTERNS).
  2. Если найдено совпадение, вернуть целевой класс персоны.
  3. Иначе использовать значение по умолчанию self.
  4. В случае child_generic/friend_generic выбрать конкретного участника группы
     по доступности/порядку в текущем состоянии заказа.

Е.6. Детектор зацикливания диалога

  1. Взять последние window (по умолчанию 4) реплики кассира.
  2. Проверить, что каждая реплика соответствует _STALL_DENIAL_RE
     (повторный отказ или переспрашивание без прогресса).
  3. Если условие истинно для всех window реплик, вернуть stalled=True и прервать цикл.

        clauses: list[dict[str, Any]] = []
        clauses.extend(_where_clauses_flat(_build_where_for_blacklist(allergens_blacklist)))
        if max_energy is not None:
            clauses.append({"energy": {"$lte": float(max_energy)}})
        if min_energy is not None:
            clauses.append({"energy": {"$gte": float(min_energy)}})
        if not clauses:
            return None
        if len(clauses) == 1:
            return clauses[0]
        return {"$and": clauses}

        if qty < 1:

        if qty > 20:

Ж.8. Детектор стагнации кассира (`dialog/pipeline.py`, `_is_stalled`)

    def _is_stalled(history: list[dict[str, str]], window: int = 4) -> bool:
        cashier_msgs = [t["text"] for t in history if t.get("speaker") == "cashier"]
        recent = cashier_msgs[-window:]
        if len(recent) < window:
            return False
        return all(_STALL_DENIAL_RE.search(m) for m in recent)
